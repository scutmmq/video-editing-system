# 技术债务与重构立项

## 1. 文档目的

本文件记录工作量较大、需独立排期的工程债务与重构项。区别于 `Feature-Proposals.md`（面向产品功能），本文件聚焦代码结构、可维护性与质量保障。小型改进项在常规迭代中直接完成，不在此列。

工作量计量：S 约 1 个工作日内，M 约 2–5 个工作日，L 约 1 周以上。

## 2. 立项项目

### TD-1　构建 / 模块体系（工作量：L）

**现状与依据**

- `src/index.html` 通过 20 余个顺序相关的 `<script>` 标签加载模块，模块间通过全局命名空间（`App`、`Upload`、`Status` 等）耦合。
- 新增功能需同时修改 `index.html` 的脚本列表与 `app.js` 的初始化逻辑，且依赖加载顺序，易出错。
- 可测试模块依赖手写 UMD 包装以兼容 Node 与浏览器，增加样板代码。

**影响**

- 扩展性受限：随着模块增多，顺序依赖与全局命名空间冲突风险上升。
- 无法使用现代依赖管理、tree-shaking、按需加载。
- 是多项大型功能（项目协作、批量处理等）的实现前置。

**方案选项**

- 方案 A：引入打包器（如 esbuild / Vite），保留现有代码风格，逐步迁移为模块导入。
- 方案 B：迁移至浏览器原生 ES Modules（`import` / `export`），不引入打包器，改造成本集中在模块边界与测试适配。

**风险**

- 影响全部源文件；需同步调整测试的模块加载方式。
- 需验证 FFmpeg.wasm、Supabase SDK、Capacitor 在新加载方式下的兼容性。

**建议**

- 在计划实施大型功能前优先评估本项。若短期不做大型功能，可暂缓。
- 倾向方案 A（打包器），对现有代码侵入较小，迁移可分阶段进行。

### TD-2　关键链路测试覆盖（工作量：M–L）

**现状与依据**

- 现有测试仅覆盖纯函数：`auth`、`jwtToken`、各处理模块的参数校验、`services`（借助 `tests/fakeSupabase.js`）、`historyMeta`。
- 无覆盖的关键链路：
  - `history.js` 的持久化编排（`_persistResult`：建任务 → 上传 → 写素材 → 完成/失败）。
  - `ffmpeg.js` 的封装（`process` / `run` / `cancel` / 清理逻辑）。
  - DOM 交互模块（upload、preview、trimRange 等）。

**影响**

- 编排与状态流转缺乏回归保护，重构或修复（如近期的 finally 清理、并发复用）易引入隐性回退。

**方案**

- `history` 编排：复用已有 `fakeSupabase`，对 `_persistResult` 的成功/失败/并发路径写单元测试。
- `ffmpeg` 封装：以 mock 的 FFmpeg 实例注入，验证 `process` 的清理在成功与异常下均执行、`cancel` 正确重置状态。
- DOM 模块：成本较高，可后置；优先覆盖纯逻辑与编排。

**风险**

- 低。主要为测试代码新增，不改动业务逻辑。

**建议**

- 与 TD-1 解耦，可独立推进。建议先覆盖 `history` 编排与 `ffmpeg` 封装两条高价值链路。
